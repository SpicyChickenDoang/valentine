00 Non‑negotiables (read before coding)

⛔
Do not improvise these rules. If any of the points below are violated, the system becomes unsafe (PHI leakage), inconsistent (wrong user context), or too slow for WhatsApp.
  1. Multi‑tenant keys everywhere. Every cache/lock/session key MUST be namespaced by tenant_id (PT / brand) and, where relevant, msisdn_id (WhatsApp number). No global agent:* keys. Each domain instance (Valentine, PizzaBot, BikeRental…) uses its own prefix.
  2. No user data in shared cache. The shared Gemini context cache is shared across users. It must contain only static content: agent system prompt + KB_ROUTER + formatting policies.
  3. User context is per‑request only. User profile, order history, media, and retrieved KB objects are built per request and sent in contents[]. Never stored in shared cache.
  4. Single‑flight cache rebuild. Redis distributed lock + SHA256 hash gate. At startup / cron, only ONE instance rebuilds the cache.
  5. Citations where domain requires traceability. For regulated / high-stakes domains (health, finance, legal) every response must include a machine‑parsable citation block listing KB object IDs + versions used. Optional for low-stakes domains (orders, bookings). Valentine (clinical) → mandatory.
  6. Human‑like delay mode by default. Latency is tuned via delay settings; disabling delays increases WhatsApp ban risk. Make “Safe / Normal / Fast” modes explicit.
  7. No sensitive user data in logs. Store only metadata (tokens/latency/KB ids/citations). For regulated domains (health → PHI, finance → PII): message/response text must be encrypted or replaced by hashes before storage.
✅
What CAN be cached safely. Static prompt policies, KB_ROUTER, templates, formatting rules, and other content that is identical for all users within the same tenant.

00d PHI/Patient Data Rule — Never store in shared cache

🔒
Rule: Never store patient-identifiable data (PHI/PII) in any shared cache (Redis, Gemini cache, memory cache). This includes: names, phone numbers, addresses, chat transcripts, symptoms, diagnoses, lab results, medications, photos, and PDFs.

Cache may include ONLY static assets:

  • System prompt / Valentine prompt
  • KB_ROUTER
  • Style templates / formatting rules
  • KB indexes (non-sensitive)

How we handle patient context instead:

  • Patient data lives in the database as Source of Truth.
  • For each request, the backend builds a minimal ephemeral context pack (in-memory per request), containing only what is necessary (e.g., allergies, current meds, last relevant labs).
  • The context pack is not reused across users, not stored in Redis/Gemini cache, and not logged.

Logging rule:

No raw message bodies in logs. Log only: metrics + trace_id + tenant_id + kb_ids + citations.

00e Cache Key Convention (mandatory)

🔑
All cache and lock keys MUST follow this pattern: 
# Cache key
cache_key = ${tenant_id}:${brand_id}:cache:v1

# Lock key
lock_key  = ${tenant_id}:${brand_id}:cache_lock

# Session key (per user)
session_key = ${tenant_id}:session:${msisdn_hash}

# State key (per job, for idempotency)

send_state_key = ${tenant_id}:send_state:${job_id}

Why: This prevents cross-tenant data leakage and makes debugging/monitoring straightforward. Every key is traceable to its tenant.

01 Architecture overview

The USE Engine AI Agent runs as a WhatsApp-facing domain agent inside the USE Engine platform. Inbound messages trigger a BullMQ job that classifies depth, selects the appropriate Gemini model, and returns a WhatsApp-formatted response. The same architecture runs any domain: clinical (Valentine), food ordering (PizzaBot), mobility (BikeRental).

🚨
Architecture constraint — the token accumulation problem. Agent system prompt (example: Valentine = 180k tokens) + full KB (example: 200 files × 3500 tokens = 700k tokens) = 880k tokens total. Loading all KB into the shared cache kills latency, ruins ROI, and causes hallucination by drowning. Even a simple pizza bot with 50 menu files runs into this. See Section 3 for the correct split.

The shared platform cache holds the agent system prompt + KB_ROUTER (typically ~2k–200k tokens depending on domain). The KB_ROUTER is a lightweight routing file (~2k tokens) that maps every user signal to the exact KB files needed — no arbitrary file count, no fixed token limit. The retriever loads only what is genuinely necessary for this user and this message. This is Sprint 1 mandatory, not optional.

flowchart TD WA[📱 WhatsApp inbound\nvia Meta Business API] --> WH[Express webhook\n/webhook/whatsapp] WH --> Q[BullMQ queue\nagent-chat-jobs] Q --> R[Router worker\nGemini 2.5 Flash-Lite\n~200 tokens] R --> C{Depth?} C -- DEPTH 0/1 --> FL[Gemini 2.5 Flash\n80% traffic] C -- DEPTH 2 --> PR[Gemini 2.5 Pro\n20% traffic] FL --> CA[Shared platform cache\nAgent prompt + KB_ROUTER] PR --> CA Q --> RET[KB Retriever\nKB_ROUTER-driven · signal-based\nonly files genuinely needed] RET --> FL RET --> PR CA --> FMT[WhatsApp formatter\nchunk / strip markdown] FMT --> WA2[📲 Send reply\nvia Meta API] FMT --> DB[(PostgreSQL\nlog session)]
💡
Key principle: cache agent prompt + KB_ROUTER — inject KB dynamically, no arbitrary limits. Agent prompt + router cached once. Full KB is never in the cache — KB_ROUTER tells the retriever exactly which files are needed for this user and this message. A pizza order query loads 0 files. A bike availability + insurance + tour recommendation query loads 2–3 files. A complex Valentine oncology + multi-system case loads as many files as clinically necessary. No cap. The only token budget that applies is on session history.

💡 Analogy — Understanding the architecture

If you've never worked with LLM context caching before, this analogy will give you the right mental model before reading the code. Examples use Valentine (clinical) because it's the hardest case — the logic applies identically to a pizza bot or a bike rental assistant.

🎓 The Cache = The agent's expertise

Valentine (clinical) is an ER physician who "learns" her 180k-token medical degree once per day. A pizza bot "learns" the menu and ordering rules. A bike agent "learns" the fleet catalog and pricing. Those tokens stay in cache all day — not re-read for every user. That's the Context Cache: loaded once, shared across all concurrent users. Cost: near zero per request.

📋 The Payload = The user's context

When a user sends a message, the system places on the agent's desk: the user's profile + exactly the KB files that match this specific request — no more, no less. The agent responds, and the files are taken back. They're never kept. That's the Dynamic Payload: rebuilt every message, never cached. The KB_ROUTER in the cache tells the retriever exactly which files to fetch. Valentine: patient dossier + clinical protocols. Pizza bot: order history + menu section. Bike rental: customer + fleet availability.

❌
What doesn't work: loading all KB files into cache for every request. A clinical agent with 200 protocol files would be slow (>5s) and would mix unrelated protocols. A pizza bot with 50 menu items cached globally would hallucinate menu combinations. Same root cause: too much noise, too few signal-matched tokens.

In code: Cache = systemInstruction of the Gemini cache (read once at startup). Payload = contents[] of the request (rebuilt every BullMQ job). KB = the retriever who picks only the right JSON files for this specific request.

🤝 Alignment: Node.js vs n8n — Real-Time vs Async

This document focuses exclusively on the left column. Both columns are complementary and part of the same CSI architecture. The boundary is strictly temporal: sub-second = native Node.js. Async / batch / tenant-configurable = n8n.

⚡ Real-Time Pipeline — Native Node.js

Scope: WhatsApp webhook → BullMQ → KB Retriever → Gemini Cache → Formatter → Patient response.

Why not n8n here? Latency is non-negotiable on WhatsApp. Managing the distributed Redis lock, targeted RAG, and the Gemini cache under 1.5s requires native code. The execution overhead of an n8n workflow on this critical inference loop is unacceptable — and the 880k token problem makes the constraint even tighter.

Sections 2–6 of this document.

🔄 Async Orchestration — n8n

Scope: Daily cache rebuild cron (already specified in Section 3), medical document / invoice generation, observability reports, new tenant onboarding, KB batch jobs.

Why n8n here? This is exactly the value of the AI Engine as defined in the CSI blueprint: configurable orchestration, tenant isolation, modifying business logic per client without redeploying the backend. No need to rebuild a workflow engine for async tasks.

CSI Architecture Blueprint — out of scope for this document.

📌
The simple rule: if the user is waiting for a response → Node.js. If nobody is waiting → n8n. Both layers coexist, neither replaces the other, and they share only one interface: Redis (BullMQ for real-time jobs, pub/sub for batch events).

const buildClassifierPrompt = (domain, history, message) => {
  const rules = DOMAIN_DEPTH_RULES[domain];
  if (!rules) throw new Error(`Unknown domain: ${domain}`);
  
  return `
You are a ${rules.name} for a WhatsApp chatbot.
Based on the conversation history and latest message,
classify the required reasoning depth:

- DEPTH_0_1: ${rules.depth_0_1}
- DEPTH_2: ${rules.depth_2}

Output ONLY one token: DEPTH_0_1 or DEPTH_2.

Conversation history (last 5 turns):
${history}

Latest message:
${message}
`.trim();
};

async function classifyDepth({ domain, history, message, geminiClient }) {
  const model = geminiClient.getGenerativeModel({ model: 'gemini-2.5-flash-lite' });
  const prompt = buildClassifierPrompt(domain, history, message);
  
  // FIX: 3s timeout — classifier must never block worker indefinitely
  const result = await Promise.race([
    model.generateContent({
      contents: [{ role: 'user', parts: [{ text: prompt }] }],
      generationConfig: { maxOutputTokens: 10, temperature: 0 }
    }),
    new Promise((_, reject) =>
      setTimeout(() => reject(new Error('CLASSIFIER_TIMEOUT')), 3000))
  ]);
  
  const raw = result.response.text().trim().toUpperCase();
  return raw.includes('DEPTH_2') ? 'pro' : 'flash';
}

module.exports = { classifyDepth, DOMAIN_DEPTH_RULES };
⚠️
Fallback rule (all domains): If classifier fails or times out (>3s), default to 'flash'. Never block the user response on classifier failure.

03 Shared platform cache

The agent system prompt + the KB_ROUTER file (~2k tokens) are cached once at server startup. The rest of the KB is never in the cache. Example: Valentine — 180k token prompt + 200 KB files × 3500 tokens = 880k total. Loading all KB into cache breaks latency, cost, and accuracy simultaneously. A pizza bot is smaller but the same principle applies. User-specific data (profile, session history, media) and the dynamically retrieved KB files are sent per-request.

🧠
Why patient data must NOT be in cache (practical reasons). Cache is built for speed, not governance. It can be dumped, replicated, logged, or accidentally shared across tenants/users. TTL is not a deletion policy. A single keying bug can mix two patients and cause clinical harm. Keep PHI in the source‑of‑truth DB and rebuild a minimal per‑request “patient context pack” each turn.
🚨
880k token problem — why full KB must NOT be in the cache.
• Latency: 880k tokens in context = TTFT >5s on Flash, >8s on Pro. WhatsApp patients assume the bot is broken after 3–4s.
• Cost: 880k cached tokens billed on every single turn — including "Bonjour". Cost ×4 vs the correct architecture.
• Hallucination: Model must scan 200 protocols × 3500 tokens to answer one question. Needle-in-haystack syndrome — it will mix NAD+ with hepatic detox protocols on ambiguous queries.
Correct split: agent prompt + KB_ROUTER cached once + KB files dynamically injected as needed = intelligence without overhead.
✅
Cache scope: agent system prompt + KB_ROUTER. TTL = 24h. A daily cron rebuilds only if SHA256 hash changed. All concurrent users share the same cache — never user-specific, never full KB files. The KB_ROUTER (~2k tokens) is the routing map that tells the retriever exactly which files to load per request.

Must contain: agent system prompt + kb/<TENANT_DOMAIN>/00_KB_ROUTER.txt.
Must NOT contain: any KB JSON object content, any user/patient data, any session history.

async function ensurePlatformCache(redis) {
  // 1. Distributed lock — UUID token, single flight across all instances
  const lockToken = crypto.randomUUID();
  // M4 fix: lock TTL 45s must be < wait ceiling 50s — was 90s vs 15s → cascade startup crash
  const acquired = await redis.set(LOCK_KEY, lockToken, 'NX', 'EX', 45);

  if (!acquired) {
    // Another instance is rebuilding — wait 50s (lock TTL 45s, so it always expires)
    for (let i = 0; i < 50; i++) {
      await new Promise(r => setTimeout(r, 1000));
      const name = await redis.get(CACHE_KEY);
      if (name) return name;
    }
    throw new Error('[CACHE] Lock held too long — rebuild did not complete');
  }

  try {
    const staticContent = buildStaticContent();
    const newHash      = sha256(staticContent);
    const storedHash   = await redis.get(HASH_KEY);
    const existingName = await redis.get(CACHE_KEY);

    // 2. Hash gate — skip rebuild if content unchanged
    if (storedHash === newHash && existingName) {
      console.log('[CACHE] Hash unchanged — reusing', existingName);
      return existingName;
    }

    // 3. Build new cache
    console.log('[CACHE] Hash changed — rebuilding');
    const displayName = `agent_platform_cache_${newHash.slice(0, 12)}`;
    // BUG-G8 FIX: timeout added — missing timeout = worker hangs indefinitely on TCP hang
    const { data } = await axios.post(
      `https://generativelanguage.googleapis.com/v1beta/cachedContents?key=${process.env.GEMINI_API_KEY}`,
      {

        displayName: displayName,
        systemInstruction: { parts: [{ text: staticContent }] },
        ttl: '86400s'
      },
      { timeout: 20000 }  // 20s — cache build only, slower than inference
    );

    // 4. Atomic swap — write name + hash in single Redis transaction
    const multi = redis.multi();
    multi.set(CACHE_KEY, data.name, 'EX', 86000);
    multi.set(HASH_KEY,  newHash,  'EX', 86000);
    await multi.exec();
    console.log('[CACHE] New cache live:', data.name);
    return data.name;

  } finally {
    // FIX-4: atomic Lua CAS — GET+DEL was a race: another instance could acquire between the two calls
    const lua = `
      if redis.call("get", KEYS[1]) == ARGV[1] then
        return redis.call("del", KEYS[1])
      else
        return 0
      end`;
    await redis.eval(lua, 1, LOCK_KEY, lockToken);
  }
}

const MAX_CHUNK = 4000; // leave margin below 4096

// Robust line-by-line table parser — handles malformed/multi-row tables safely
function markdownTablesToText(input) {
  const lines = input.split('\n');
  let out = [], i = 0;
  const isTableRow = l => /\|/.test(l) && l.trim().startsWith('|');
  const isDivider  = l => /^\|\s*:?[-]{3,}/.test(l.trim());
  while (i < lines.length) {
    if (!isTableRow(lines[i])) { out.push(lines[i]); i++; continue; }
    const header = lines[i].split('|').map(c => c.trim()).filter(Boolean); i++;
    if (i < lines.length && isDivider(lines[i])) i++; // skip separator
    out.push('', '*Table*');
    while (i < lines.length && isTableRow(lines[i]) && !isDivider(lines[i])) {
      const cells = lines[i].split('|').map(c => c.trim()).filter(Boolean);
      out.push(`• ` + header.map((h, j) => `${h}: ${cells[j] || '—'}`).join(' | '));
      i++;
    }
    out.push('');
  }
  return out.join('\n');
}

function formatForWhatsApp(text) {
  // 1. Convert tables BEFORE any other processing (never drop clinical data)
  text = markdownTablesToText(text);
  return text
    .replace(/<[^>]+>/g, '')           // strip HTML
    .replace(/#{1,6}\s/g, '*')         // h1-h6 → bold
    .replace(/\*\*(.*?)\*\*/g, '*$1*') // **bold** → *bold*
    .replace(/__(.*?)__/g, '_$1_')    // __italic__ → _italic_
    .replace(/^\s*[-*+]\s/gm, '• ')   // normalize bullets
    .replace(/\n{3,}/g, '\n\n')       // max 2 consecutive newlines
    .trim();
}

function chunkMessage(text, max = MAX_CHUNK) {
  if (text.length <= max) return [text];

  const chunks = [];
  let remaining = text;

  while (remaining.length > max) {
    // find last sentence boundary before limit
    let cut = remaining.lastIndexOf('. ', max);
    if (cut === -1) cut = remaining.lastIndexOf('\n', max);
    if (cut === -1) cut = max; // hard cut
    chunks.push(remaining.slice(0, cut + 1).trim());
    remaining = remaining.slice(cut + 1).trim();
  }
  if (remaining) chunks.push(remaining);
  return chunks;
}

async function sendChunked(waClient, to, rawText) {
  const formatted = formatForWhatsApp(rawText);
  const chunks = chunkMessage(formatted);
  for (const [i, chunk] of chunks.entries()) {
    await waClient.sendMessage({ to, body: chunk });
    if (i < chunks.length - 1) await sleep(150); // typing feel
  }
}

module.exports = { formatForWhatsApp, chunkMessage, sendChunked };

07 Function Calling — Deterministic Tools

The golden rule of production AI: a LLM must never do clinical math, calculate prices, or check availability. The model is the "mouth" — the Node.js server must be the "logical brain". We use Gemini's tools parameter to give it access to strict, deterministic calculators. Gemini extracts the data, pauses its response, asks Node.js to run the exact computation, receives the JSON result, and then drafts the final WhatsApp message.

🚫
Why the LLM must never compute. A model calculating HOMA-IR, eGFR, a pizza price with promo code, or a bike rental duration will hallucinate. Not sometimes — always, at scale. The math looks plausible, the result is quietly wrong. Clinical context (Valentine): patient safety issue. Commerce context (pizza/bike): billing error. Function Calling is the fix: the model identifies what to compute, Node.js computes how, deterministically.

🩺 Example: Lab Calculator (Valentine / health domain)

Model reads lab PDF/image, extracts biomarkers, asks Node.js to compute clinical ratios (HOMA-IR, TyG, eGFR…) via exact formulas. Same pattern: pizza → price_calculator with promo rules; bike → availability_checker against fleet DB.

// services/labCalculator.js
function calculateLabRatios(markers, context = {}) {
  const results = {};

  // HOMA-IR = (fasting glucose mmol/L × fasting insulin µIU/mL) / 22.5
  if (markers.FBG && markers.fasting_insulin) {
    const fbg_mmol = markers.FBG_unit === 'mg/dL' ? markers.FBG / 18.0 : markers.FBG;
    results.HOMA_IR = round2((fbg_mmol * markers.fasting_insulin) / 22.5);
  }

  // TyG index = ln(TG mg/dL × FBG mg/dL / 2)
  if (markers.TG && markers.FBG) {
    const tg_mgdl  = markers.TG_unit  === 'mmol/L' ? markers.TG  * 88.57 : markers.TG;
    const fbg_mgdl = markers.FBG_unit === 'mmol/L' ? markers.FBG * 18.0  : markers.FBG;
    results.TyG_index = round2(Math.log(tg_mgdl * fbg_mgdl / 2));
  }

  // eGFR — CKD-EPI 2021 race-free (KDIGO 2021) — ckdEpi2021 defined below calculateLabRatios
  if (markers.creatinine && context.age) {
    results.eGFR = ckdEpi2021(markers.creatinine, context.age, context.sex);
  }

  results.computed_at = new Date().toISOString();
  return results;
}

function round2(n) { return Math.round(n * 100) / 100; }

// BUG-G6 FIX: CKD-EPI 2021 race-free formula — was called but never defined → ReferenceError on every renal panel
function ckdEpi2021(creatinine_mgdl, age, sex = 'M') {
  const kappa = sex === 'F' ? 0.7  : 0.9;
  const alpha = sex === 'F' ? -0.241 : -0.302;
  const scr_k = creatinine_mgdl / kappa;
  const base  = sex === 'F' ? 1.012 : 1.0;
  return round2(
    142 * Math.pow(Math.min(scr_k, 1), alpha)
        * Math.pow(Math.max(scr_k, 1), -1.200)
        * Math.pow(0.9938, age)
        * base
  );
}

08 User Memory Architecture

The core problem: Redis sessions expire after 1h of inactivity. Without a persistence layer, the agent wakes up with amnesia every time a user returns — re-asking their name, preferences, order history. This is a bad experience for any domain. Valentine (clinical) makes this a safety issue: re-asking allergies or active medications on a returning patient violates clinical constraints. The architecture below fixes this permanently.

🚫
What happens today without this system: Redis TTL expires → worker loads empty context → agent responds as if it's a new user → asks for information already given → user irritated, experience broken. For Valentine: asks for name, location, symptoms already given — safety issue, not just UX. This is the default failure mode.

// Three memory layers

🗄️ Hard Memory — PostgreSQL (permanent)

Domain-specific persistent facts. Valentine: name, age, sex, language, allergies, conditions, medications, labs, terrain flags. Pizza bot: name, address, dietary preferences, favourite orders. Bike rental: name, rental history, preferred bike type. Loaded automatically at every new session. Never expires. Never re-asked.

📋 Soft Memory — PostgreSQL (accumulates)

Session summaries (3–5 lines each). Last 3 sessions injected at context rebuild. Generated by N8N + Gemini Flash-Lite at conversation close. Valentine: chief complaint, recommendations, follow-up notes. Pizza bot: last order summary, delivery feedback. Bike rental: last trip, return condition notes.

⚡ Session Cache — Redis (volatile, TTL 1h, tenant‑scoped)

8k token-aware history. Keyed by tenant_id + phone (and msisdn if needed). Active conversation fluidity. Disappears after 1h inactivity. This is a comfort cache, not the source of truth. When it expires, PostgreSQL rebuilds it silently. The user never sees the difference.

// ─── INVARIANT 1 ─── Canonical ID ──────────────────────────────────────────
// Input : any filename stem (mixed case, spaces, hyphens OK)
// Output: stable lowercase slug WITHOUT type-token
// e.g. "32_IV_IVC_HIGH_DOSE_Monograph_v3" → "32_iv_ivc_high_dose_v3"
// Rule: type tokens (monograph, protocol, policy) live in the JSON "type" field, NOT the id.
const TYPE_TOKENS = ['monograph', 'protocol', 'policy'];  // stripped from id slug
function canonicalId(stem) {
  let slug = stem
    .toLowerCase()
    .replace(/[\s\-]+/g, '_')   // spaces/hyphens → underscore
    .replace(/[^a-z0-9_]/g, ''); // strip everything else
  for (const tok of TYPE_TOKENS)
    slug = slug.replace(new RegExp(`_${tok}(?=_|$)`, 'g'), '');
  return slug.replace(/_{2,}/g, '_').replace(/^_|_$/g, ''); // clean artefacts
}

// Section numbers 0-5 map to normalised key names.
// VISIBILITY is read from the SECTION header line only — never from the body.
const SECTION_KEYS = {
  '0': 's0_quick_reference_internal',
  '1': 's1_claims_discipline_internal',
  '2': 's2_patient_communication',  // suffix added after VISIBILITY check
  '3': 's3_pharmacokinetics_adme_internal',
  '4': 's4_limitations_failure_modes_internal',
  '5': 's5_clinical_decision_trees_internal',
};

function parseSections(raw, file) {  // file passed for error context
  const content = {};
  // Match: "=== SECTION 2 [— ANYTHING] [PATIENT_OK] ===" followed by body until next === or EOF
  // C1 fix: \Z is Python-only — in JS use (?![\s\S]) to match end-of-string
  const re = /^={3,}[ \t]*SECTION[ \t]+(\d+)([^\n]*)\n([\s\S]*?)(?=^={3,}|(?![\s\S]))/gm;
  let m;
  while ((m = re.exec(raw)) !== null) {
    const num      = m[1];
    const header   = m[2];                         // rest of header line
    const isPublic = header.includes('PATIENT_OK');  // ONLY from header, never body
    const body     = m[3].trim();
    let key        = SECTION_KEYS[num];
    if (!key) { console.warn(`  ⚠ Unknown section ${num} — skipped`); continue; }
    // ─── Compile-time VISIBILITY guards (throw = blocks deploy) ──────────
    if (isPublic && num !== '2') {
      throw new Error(
        `[migrateKB] VISIBILITY ERROR in ${file}: [PATIENT_OK] on SECTION ${num} — ` +
        'only SECTION 2 (patient communication) may be public. Fix the .txt file.'
      );
    }
    if (num === '2' && !isPublic) {
      throw new Error(
        `[migrateKB] VISIBILITY ERROR in ${file}: SECTION 2 is missing [PATIENT_OK] — ` +
        'patient-facing section must be explicitly marked. Add [PATIENT_OK] to the SECTION 2 header.'
      );
    }
    // Section 2 suffix depends on header flag; all others have fixed suffix already
    if (num === '2') key += isPublic ? '_public' : '_internal';
    content[key] = body;
  }
  // MIN-3: require SECTION 0 (quick_ref) and SECTION 2 (patient_comms) — silently incomplete KB is a clinical hazard
  const REQUIRED_SECTIONS = ['0', '2'];
  for (const num of REQUIRED_SECTIONS) {
    if (!Object.keys(content).some(k => k.startsWith(`s${num}_`)))
      throw new Error(`[migrateKB] MISSING required SECTION ${num} in ${file}`);
  }
  return content;
}

⚡ Dual-Mode Routing — Fast-Lane <2s

Routing is two-stage. Node runs first (deterministic, in-memory, <5ms), then the LLM router finalizes using KB_ROUTER from cache. The two stages are never swapped. The LLM never sees a request before the prefilter has run.

Stage 1 — Node prefilter

Deterministic. In-memory Map scan. O(N) on trigger count — <5ms wall-clock (N = 20–50 triggers; <1ms in practice). Never reads disk, never calls network. Checks hard-stop triggers against the incoming message. If triggered, adds specific policy IDs to the fetch set — no LLM involved.

Stage 2 — LLM router (cached)

Gemini Flash-Lite reads KB_ROUTER (in cache). Receives: user message + prefilter summary. Returns exactly: {"always_load":[], "fetch":[], "reasons":[]}. No free text. Temperature 0.

  const prefilterHits = runPrefilter(message);

  // Stage 1b — tag-match prefilter (sync, no LLM, feeds Stage 2)
  // Scans index in-memory, returns top candidates by tag overlap score. Max 30.
  // F1: hoist toLowerCase() outside the map — 1 call, not N×M calls
  const needle = message.toLowerCase();
  const tagCandidates = [...getIndex().entries()]
    .map(([id, e]) => ({ id, score: e.tags.filter(t => needle.includes(t.toLowerCase())).length }))
    .filter(c => c.score > 0)
    .sort((a, b) => b.score - a.score)
    .map(c => c.id);  // top candidates — slice(0,30) applied in prompt

  // Stage 2 — LLM finalizes (~200ms)
  const llm = await runLLMRouter(message, prefilterHits, tagCandidates, cacheName);

  // Stage 3 — Merge rule (source of truth)
  // Final = ALWAYS_LOAD ∪ prefilter_hits ∪ llm.always_load ∪ llm.fetch
  const finalSet = new Set([
    ...ALWAYS_LOAD,
    ...prefilterHits,
    ...(llm.always_load || []),
    ...(llm.fetch       || []),
  ]);
  return [...finalSet];  // kbRetriever.loadKBFiles() resolves via index.json
}

function get(id) {
  const entry = getIndex().get(id);
  if (!entry) { console.warn(`[kbRetriever] id not found in index: ${id}`); return null; }
  // entry.filepath is relative to repo root (e.g. "kb/json/32_iv_ivc_high_dose_v3.json")

// ─── Batch loader (called by chatWorker after KB_ROUTER returns ids) ────────
// Hard caps: prevent a single request from loading 40+ files and blowing token budget.
// MAX_KB_FILES: max number of KB objects per request. MAX_KB_CHARS: total content chars.
// If exceeded: log warning + truncate (never throw — agent degrades gracefully).
const MAX_KB_FILES = 15;           // tune per domain (Valentine: 15, pizza: 5)
const MAX_KB_CHARS = 120_000;      // ~30k tokens at ~4 chars/token

function loadKBFiles(fileIds) {
  if (!fileIds?.length) return [];
  const loaded = fileIds.slice(0, MAX_KB_FILES).map(get).filter(Boolean);
  if (fileIds.length > MAX_KB_FILES)
    console.warn(`[kbRetriever] capped at ${MAX_KB_FILES} files (requested ${fileIds.length})`);
  // Total chars guard
  let chars = 0;
  return loaded.filter(f => {
    const sz = JSON.stringify(f).length;
    if (chars + sz > MAX_KB_CHARS) { console.warn(`[kbRetriever] MAX_KB_CHARS reached, dropping ${f.id}`); return false; }
    chars += sz; return true;
  });
}

function parseCitedObjects(responseText) {
  // Matches: *Protocols referenced: protocol_nad_iv_v2, protocol_bali_belly_v3*

  // FIX: null guard — parseCitedObjects can return undefined on malformed response
  retrieved = retrieved || [];
  cited = cited || [];

  if (!retrieved.length) return true;   // nothing loaded = N/A = OK
  if (!cited.length) return false;     // KB loaded, model cited nothing = miss
  return retrieved.every(id => cited.includes(id));
}

const PROFILE_KEYS = [
  'allergies',       // string[]
  'conditions',      // string[]
  'medications',     // {name, dose, prescriber}[]
  'last_labs',       // {biomarker: value}
  'terrain_flags',   // {axis: color}
  'follow_up_date',  // YYYY-MM-DD
  'follow_up_note',  // string
  'track'            // 'A' | 'B'
];

/**
 * Extract profile update JSON from LLM response text
 * Returns null if no update block found or parsing fails
 * 
 * Expected format in LLM response:
 * ```json
 * {"allergies": ["penicillin"], "last_labs": {"TSH": 2.81}}
 * ```
 * 
 * @param {string} responseText - Full LLM response including JSON block
 * @returns {Object|null} - Parsed profile update or null
 */
function extractProfileUpdate(responseText) {
  if (!responseText || typeof responseText !== 'string') {
    return null;
  }

  // Pattern 1: ```json ... ``` block (most common)
  const jsonBlockMatch = responseText.match(/```json\s*([\s\S]*?)\s*```/i);
  
  // Pattern 2: {"profile_update": ...} inline (fallback)
  const inlineMatch = responseText.match(/\{[\s\S]*"(allergies|conditions|medications|last_labs|terrain_flags)"[\s\S]*\}/);

  const jsonStr = jsonBlockMatch?.[1] || inlineMatch?.[0];
  
  if (!jsonStr) {
    return null;  // No profile update in this response
  }

  try {
    const parsed = JSON.parse(jsonStr);
    
    // Validate: must have at least one known profile key
    const hasValidKey = PROFILE_KEYS.some(key => key in parsed);
    if (!hasValidKey) {
      console.warn('[jsonExtractor] JSON found but no valid profile keys');
      return null;
    }

    // Sanitize: only return known keys (prevent injection of arbitrary fields)
    const sanitized = {};
    for (const key of PROFILE_KEYS) {
      if (parsed[key] !== undefined) {
        sanitized[key] = parsed[key];
      }
    }

    // Type validation for critical arrays
    if (sanitized.allergies && !Array.isArray(sanitized.allergies)) {
      sanitized.allergies = [sanitized.allergies];  // Wrap single value
    }
    if (sanitized.conditions && !Array.isArray(sanitized.conditions)) {
      sanitized.conditions = [sanitized.conditions];
    }

    return sanitized;
    
  } catch (err) {
    console.warn('[jsonExtractor] JSON parse failed:', err.message);
    return null;  // Malformed JSON — skip update, don't crash
  }
}

/**
 * Extract WhatsApp-safe text (strip the JSON block for user response)
 * @param {string} responseText - Full LLM response
 * @returns {string} - Clean text for WhatsApp
 */
function stripProfileBlock(responseText) {
  if (!responseText) return '';
  return responseText
    .replace(/```json[\s\S]*?```/gi, '')
    .replace(/\n{3,}/g, '\n\n')
    .trim();
}

  ROUND(AVG(cached_tokens::numeric / NULLIF(input_tokens,0)) * 100, 1)
                                                           AS cache_hit_pct,
  ROUND(AVG(model_latency_ms))                             AS avg_latency_ms,
  PERCENTILE_CONT(0.95) WITHIN GROUP (ORDER BY model_latency_ms)
                                                           AS p95_latency_ms,
  SUM(CASE WHEN safe_mode THEN 1 ELSE 0 END)              AS safe_mode_count,
  SUM(CASE WHEN citation_match = FALSE THEN 1 ELSE 0 END)  AS citation_misses,
  ROUND(AVG(wa_chunks_sent), 1)                           AS avg_wa_chunks
FROM chat_logs
WHERE created_at >= NOW() - INTERVAL '24 hours'
GROUP BY 1
ORDER BY 1 DESC;

⚠ Watch out — Critical pitfalls

These three points are the most common integration mistakes. Read before writing any code.

🔒
Data Security — Patient data leaking into the shared cache Strict rule: the patient dossier and session history must never be included in the shared cache. The cache is a resource shared across all platform users. Only the static agent system prompt + KB_ROUTER goes in cache. Everything user-specific goes only in the dynamic contents payload. For Valentine: patient dossier + labs = PHI, especially critical. For any domain: wrong user context = broken UX or billing error.
⏱️
Media Expiry — WhatsApp images expire in 5 minutes The webhook receives a media_id, not the image itself. This media_id must be resolved via the Meta Graph API to get a download URL, then converted to base64 — all inside the worker, immediately after the job is triggered. Never store the URL in the database or pass it deferred: it will have expired. If the worker is waiting in the queue > 5 min, the fetch will fail silently.
🔄
Fallback — If Gemini Pro is unavailable (429 / 503) Implement safe mode, not degraded mode: if Pro returns 429/503, retry 2× with exponential backoff (1s, 3s). If still unavailable on a DEPTH_2 case (multi-system, labs, images) — do not silently fall back to Flash. Real risk: Flash can generate an "overconfident" response on a complex case. Correct behavior: respond with the domain-appropriate safe-mode message (see Section 0 Non-negotiable #5). Then call notifyAlert(tenantId, { type: 'pro_unavailable', msisdnHash, job_id }). This function POSTs a JSON payload to process.env.ALERT_WEBHOOK_URL — set this per tenant in the env (Slack incoming webhook, WA internal relay, or email endpoint). It is non-blocking and fail-silent: a failed alert must never block the user response. Log pro_unavailable: true, safe_mode: true. Flash only if a second DEPTH classification returns 0/1. 
// utils/notifyAlert.js
const axios = require('axios');
async function notifyAlert(tenantId, payload) {
  const url = process.env.ALERT_WEBHOOK_URL;  // set per tenant
  if (!url) return;  // silent if not configured
  try {
    await axios.post(url, { tenantId, ...payload, ts: new Date().toISOString() }, { timeout: 5000 });  // BUG-G8 FIX: alert webhook must not block
  } catch (e) {
    console.warn('[notifyAlert] failed — non-blocking:', e.message);
  }
}

TRAP #2 — Token expires in 24h (the fatal mistake)

Le "Temporary Access Token" de la page "Getting Started" Meta expire dans 24 heures. L'app crashera demain sans raison apparente.

Permanent fix:
1. Business Manager → System Users → Ajouter un utilisateur Admin
2. Clic sur "Generate New Token"
3. Coche les 2 permissions : whatsapp_business_messaging + whatsapp_business_management
4. Ce token (commence par EAA...) n'expire jamais. Mets-le dans .env.
